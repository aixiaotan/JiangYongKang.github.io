Spring Data Jpa 对象关联与加载策略

发表于 2018-04-07 | 分类于 后端开发 |

Spring Data Jpa 是对 Hibernate 做的一层薄封装，使我们能够更加快速的开始一个项目，专注于业务逻辑。大大的减少了繁琐的配置。所以 Jpa 中的对象关联，加载机制，查询方法等等，和 Hibernate 的使用大同小异。基本上如果使用过 Hibernate，那么可以很平滑的过渡到 Jpa 上，本文将结合 Jpa 对他们的对象关联，加载策略进行讲解。

如何集成?

在 SpringBoot 中集成 Jpa 非常简单，只需要加上以下依赖，并添加关键的几个配置项，就可以将 Jpa 整合进来。

级联操作与加载策略

关联对象之间默认是没有级联操作的，通过 cascade 属性可以设置级联操作，它的取值范围是:

  • CascadeType.ALL 包含以下所有
  • CascadeType.PERSIST 只有主对象在新增操作的时候，才会保存或更新关联对象
  • CascadeType.MERGE 指主对象新增或者变化，关联对象也会新增或者变化
  • CascadeType.REMOVE 删除主对象的同时删除关联对象
  • CascadeType.REFRESH 保存主对象时级联保存关联对象
  • CascadeType.DETACH 删除对象时，如果该对象有外键无法删除，它会撤销所有相关外键

加载策略有两种，立即加载和延迟加载，默认情况下是延迟加载。可以通过 fetch 属性来设置加载策略，它的取值范围是：

  • FetchType.LAZY 延迟加载，只有调用了关联对象的时候才进行数据库查询
  • FetchType.EAGER 立即加载，在查询主对象的同事，会将关联对象也从数据查询出来

序列化

在对实体类进行序列化时，往往会有意向不到的事情发生，比如想要的属性出不来，不想要的属性加载一大堆，甚至属性循环加载造成死循环。这一部分就如何随心所欲控制序列化进行讲解。
通常在项目中，每一个实体类多多少少都会有关联对象的存在，比如我们需要查询一个用户的列表，我们期望的是显示用户的所有属性，而用户所关联的账户我们不需要显示。百度了一下，有多种方法可以做到。

  • 在不需要显示的属性上添加 @JsonIgnore 注解，这种方法的确可以达到我们的预期，但是如此一来，所有的用户对象都不在返回被忽略的属性。这种做法局限性太强，不可能适用于所有接口，假如另一个接口需要返回的是用户列表和用户对应的账户，那么这种做法会让你左右为难。
  • 使用 Jackson 的 MixInAnnotation 特性，加之 AOP 实现动态过滤，这种做法繁琐不说，性能上会存在问题。因为这种做法会用到大量的反射，每次接口调用都存在动态解释。
  • 创建一个新的类，包含用户和该用户的账户，这种做法的确可以达到目的，但是可维护性太差，业务逻辑比较复杂的情况，会创建很多诸如此类的类。

SpringBoot 在 对象 JSON 互转方面使用的是 Jackson，这个第三方库在对一个对象进行序列化的时候，会调用该对象的全部属性方法（包括关联对象的属性方法）一层一层的转换。也就是说，对象如果做了双相关联，那么这里是肯定会出现死循环的。那么如何实现按需加载呢？
